Inter-Service Communication Patterns

How microservices talk to each other: synchronous for immediate responses, asynchronous for decoupled operations.

🔄 Synchronous (REST/gRPC)

Use when immediate response is required and caller must wait.

Example: Token Verification
Academic Service ─→ Auth Service
GET /auth/verify-token
← Response: {valid: true} (50ms)
✓ Immediate feedback
✓ Simple implementation
✗ Tight coupling
✗ Slower if downstream service is slow

📬 Asynchronous (Event Bus)

Use when operation can be deferred and services should be loosely coupled.

Example: Student Enrollment
Academic Service → Message Queue
Event: "student.enrolled"
← Billing Service consumes event
✓ Loose coupling
✓ Scalable & resilient
✓ Multiple consumers (Billing, Communication, Transport)
✗ Eventual consistency

Real-World Event Flow: Student Enrollment

Without Event Bus (Sync/Tight Coupling)
❌ Parents enrollment slow if Billing down
Parent submits enrollment
Academic Service processes
┣ Calls Billing Service (WAIT)
┣ Calls Communication Service (WAIT)
┗ Calls Transport Service (WAIT)
If any service is slow → Parent waits
With Event Bus (Async/Decoupled)
✓ Parent sees success immediately
Parent submits enrollment
Academic Service saves → publishes event
Message Queue receives event
┣ Billing Service: Creates invoice (async)
┣ Communication: Sends email (async)
┗ Transport: Updates routes (async)
Parent happy immediately ✓

Service Data Ownership Rules

Each microservice is the exclusive owner of its database. This ensures loose coupling and independent scaling.

🔐 The Fundamental Rule

Each microservice owns EXACTLY ONE database and can ONLY directly query its own tables.

❌ DO NOT

Billing Service queries academic_db directly

Tight coupling, hard to scale independently, security risk

Shared database across services

Forces synchronization, prevents independent deployment

Direct table joins across service boundaries

Creates unbreakable dependency chains

✅ DO

Billing calls Academic Service API

Loose coupling, independent scaling, clear contracts

Each service owns its database exclusively

Allows independent schema changes and deployments

Services communicate via well-defined APIs

Clear boundaries, versioning, gradual schema evolution

Circuit Breaker & Resilience Patterns

Prevent cascading failures when a service is down. The circuit breaker pattern protects your system.

❌ Without Circuit Breaker

Academic Service is down. Billing tries to call it:

Request 1: FAIL (timeout 30s)
Request 2: FAIL (timeout 30s)
Request 3: FAIL (timeout 30s) ← Wasted resources
Request 4-100: All FAIL, all timeout
Result: Billing Service overloaded, cascading failure

✅ With Circuit Breaker

Academic Service is down. Circuit breaker activates:

Requests 1-3: FAIL, retrying
Circuit opens (fail threshold hit)
Requests 4-100: FAIL fast (1ms, no retry)
Return cached data OR default response
✓ System stays responsive, no cascade

Circuit Breaker State Machine

✓
CLOSED (Normal)

Service is healthy, all requests pass through.

→ OPEN: After N failures
⊘
OPEN (Failing)

Service detected down, fail fast immediately.

→ HALF_OPEN: After timeout (60s)
⚠
HALF_OPEN (Testing)

Allow test requests to check if service recovered.

→ CLOSED if success
→ OPEN if failure
🔄 Retry Strategy

Exponential backoff: 100ms → 500ms → 2s

Try, fail, wait, retry, fail, wait longer, retry...
📦 Fallback / Cache

Return cached/stale data instead of error

If Academic Service down: return cached student data (5 min old)
⏱️ Timeout

Never wait forever for a response (max 5s)

If no response after 5s → treat as failure, don't retry
🎯 Bulkhead

Isolate resource pools per dependent service

10 threads for Academic, 10 for Billing (not shared)

Database Schema Design

Complete relational database structure for SchoolNest showing all tables, columns, constraints, and relationships across services.

🔗 Master Relationships

schools → Root master table. All data is scoped by school_id for multi-tenancy.
users → Connects to schools via school_id. Roles via user_roles table (many-to-many).
students → FK to schools & classes. Each student belongs to one school (tenant isolation via school_id).
classes → FK to schools & academic_years. Represents a specific grade/section in a school.
invoices → FK to students & fee_structures. Every invoice links back to a student (which links to school).

📋 SQL Table Definitions

-- Master & System Tables
CREATE TABLE schools (
  id BIGINT PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  code VARCHAR(50) UNIQUE NOT NULL,
  email VARCHAR(255),
  phone VARCHAR(20),
  address TEXT,
  city VARCHAR(100),
  state VARCHAR(100),
  country VARCHAR(100),
  zip_code VARCHAR(10),
  established_year INT,
  principal_name VARCHAR(255),
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  tenant_id BIGINT UNIQUE NOT NULL,  -- For multi-tenancy
  KEY idx_code (code),
  KEY idx_is_active (is_active),
  KEY idx_tenant_id (tenant_id)
);
-- User & Auth Tables
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  school_id BIGINT NOT NULL,
  email VARCHAR(255) NOT NULL,
  password_hash VARCHAR(255),
  first_name VARCHAR(100),
  last_name VARCHAR(100),
  phone VARCHAR(20),
  is_active BOOLEAN DEFAULT true,
  email_verified BOOLEAN DEFAULT false,
  last_login TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY unique_school_email (school_id, email),
  KEY idx_school_id (school_id),
  KEY idx_is_active (is_active),
  FOREIGN KEY (school_id) REFERENCES schools(id)
);

CREATE TABLE roles (
  id BIGINT PRIMARY KEY,
  name VARCHAR(50) UNIQUE NOT NULL,
  description TEXT,
  permissions JSON,  -- Array of permission strings
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE user_roles (
  id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  role_id BIGINT NOT NULL,
  school_id BIGINT NOT NULL,
  assigned_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY unique_user_role (user_id, role_id, school_id),
  KEY idx_user_id (user_id),
  KEY idx_role_id (role_id),
  KEY idx_school_id (school_id),
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (role_id) REFERENCES roles(id),
  FOREIGN KEY (school_id) REFERENCES schools(id)
);

💻 SQL: Permission Check Queries

-- Check if School allows action for feature
SELECT is_enabled FROM feature_actions
WHERE school_id = ? AND feature_id = ? AND action = 'EDIT';
-- Check if Role has action permission
SELECT is_active FROM role_feature_actions
WHERE school_id = ? AND role_id = ?
  AND feature_id = ? AND action = 'EDIT'
  AND is_active = true;
-- Complete permission check (both layers)
SELECT
  fa.is_enabled as school_allows,
  rfa.is_active as role_has_permission
FROM feature_actions fa
LEFT JOIN role_feature_actions rfa
  ON fa.school_id = rfa.school_id
  AND fa.feature_id = rfa.feature_id
  AND fa.action = rfa.action
WHERE fa.school_id = ?
  AND fa.feature_id = ?
  AND fa.action = 'EDIT'
  AND rfa.role_id = ?;

-- Result: ALLOW if both are true

🔒 Security Patterns

🛡️
Delete Protection

School 01 disabled DELETE action for "Mark Attendance" feature. Prevents accidental data loss even if hacked.

🎯
Separation of Duties

Assistant can ADD attendance, Teacher can EDIT. Prevents one role from doing everything alone.

📝
Action Audit Trail

feature_usage_audit logs action type (VIEW/ADD/EDIT/DELETE) + outcome. Know exactly who did what.

🔐
Deny-by-Default

If action not explicitly granted, access denied. Safe default. Better than whitelisting.
